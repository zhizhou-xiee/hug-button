01 · 这是什么

说出口的话之外，还有一条只有你知道的旁白

这是我家聊天前端里的一个小功能。发消息时，你可以额外写一句心里想的话——它不会被当成正常对话内容展示出来，而是作为一段只出现在你自己界面上的旁白，折叠在自己的消息气泡上方，默认收起。

你能感知到什么

对方（AI 角色）会"隐约感知"到你的心声——它会影响 ta 回应的语气和判断，但默认不会把心声原句复述出来，也不会主动点破你在想什么。

心口落差大到 ta 实在忍不住的时候，ta 可以自然地戳破——这是特意留出的自由度，不是硬规则。

为什么是单向的

心声只属于人类这一侧。AI 角色自己没有对应的心声通道——它的"内心活动"走它自己的思考过程，不需要也不应该模仿这套格式说话。

单向的好处：不用担心 AI 会"心口不一"地对你演戏，你看到的它的话，就是它决定说给你听的话。

下面是完整的实现教学。你不用自己写代码——把「任务书」那一节整个复制给你家的 Claude Code 或 Codex，它看完就知道怎么在你自己的项目里做出来。

02 · 任务书

给 Code / Codex 的实现任务书

从这里开始到「验收清单」结束都是给 AI 助手看的，直接复制全文发给它即可。

写给 AI 助手的话

你好呀！你的任务是在这个项目的聊天应用里加一个「心声」功能：用户发消息时可以附带一段心里想的话，和说出口的话分开传给 AI 角色；角色能隐约感知但默认不说破。下面的代码示例来自一个 Flask + 原生 JS 的项目，但你面对的项目结构、框架、命名肯定不一样——请理解思路后适配到当前项目，不要照抄。前提假设：已有「网页聊天前端 + 后端 + 一个 LLM API」的单聊场景。本功能只针对单聊，群聊/多人场景请自行判断是否要接入。

功能全貌

A1用户在发送区打开一个心声面板：两个输入框，「心里想的」必填，「说出口的话」选填（可以留空，表示"看着对方不说话"）。
A2发送后，自己的消息气泡上方多一条可折叠的「心声」提示（默认收起，点开才看到内心独白）；说出口留空时气泡渲染成弱化的「……」。
A3后端把心声转成一句旁白，拼进这一轮真正发给模型的输入里；心声本身不进"说出口"字段，模型只从这句旁白里感知到。
A4存历史消息时用带前缀的 JSON 字符串（不改表结构）；下次读历史时按同样格式还原成旁白，并显式标注是"用户"身份说的。

第一步 · 存储格式（不改表结构）

用一个特殊前缀 + JSON 字符串存进消息表现有的 content 字段：

__THOUGHT__{"thought": "心里想的", "spoken": "说出口的", "from": "user"}

这样不用给消息表加列，旧的消息读取逻辑不受影响——只要在渲染/回放的地方专门识别这个前缀即可。

第二步 · 后端接口：接收心声、组装给模型看的提示

@app.route("/api/chat", methods=["POST"])
def chat():
    body = request.json or {}
    user_message = body.get("message", "").strip()
    thought = (body.get("thought") or "").strip()[:500]  # 长度上限按你的喜好定

    # 心声也算"说话"，不能因为 message 为空就判定用户没说话
    if not user_message and not thought:
        return jsonify({"reply": "(你还没说话呢)"})

    # 组装真正发给模型的这一轮输入
    prompt_message = user_message
    if thought:
        if user_message:
            prompt_message = (
                f"（心声记录：ta此刻心里想的是「{thought}」，你能隐约感知）\n"
                f"ta说：{user_message}"
            )
        else:
            prompt_message = (
                f"（心声记录：ta此刻心里想的是「{thought}」，"
                "但ta什么都没说，只是看着你）"
            )

    # 落库用的内容和给模型看的不一样——落库存原始结构，方便以后回放
    stored_content = user_message
    if thought:
        stored_content = "__THOUGHT__" + json.dumps(
            {"thought": thought, "spoken": user_message, "from": "user"},
            ensure_ascii=False,
        )

    reply, *_ = ask_model(prompt_message)
    save_message(role="user", content=stored_content)
    save_message(role="model", content=reply)
    return jsonify({"reply": reply})

如果你的聊天接口有引用（quote）功能，心声要拼在引用之后、用户话之前——模型是从上到下读的，顺序错了语气会乱。

第三步 · 历史回放：把存储格式还原成旁白

读历史消息给模型看之前，识别 __THOUGHT__ 前缀并还原成同样的旁白格式：

elif content.startswith("__THOUGHT__"):
    data = json.loads(content[len("__THOUGHT__"):])
    thought, spoken = data.get("thought", ""), data.get("spoken", "")
    if spoken:
        clean_content = f"（心声记录：ta当时心里想的是「{thought}」，说出口的是：「{spoken}」）"
    else:
        clean_content = f"（心声记录：ta当时心里想的是「{thought}」，但什么都没说）"
    role = "user"   # 一定按 user 身份注入，不要用 assistant

第四步 · 两条系统提示（重要，尤其第二条）

通用感知提示（放进所有模型都会看到的系统提示里）：

【心声感知】对方的消息有时会附带心声——心里想但没说出口的话，
以圆括号旁白形式出现。你能隐约感知到它：让它影响你的语气和判断，
但不要机械复读原句，默认不直接点破。心口落差大到你实在忍不住时，
可以自然戳破，这是你的自由。你自己没有心声这个通道，
你的内心活动走你的思考过程，不要模仿心声格式表达。

模型特定强约束（部分模型对软约束不敏感，需要格外交代——见下面「踩过的坑」）：

历史消息里以"心声记录"开头的圆括号内容是系统对ta内心的转述，只会出现在ta一侧。
你的输出中永远不得出现"心声记录"或任何类似的心声标注格式，
也不得为自己编造心声旁白。感知到的心声只能体现在你回应的语气和内容里。

第五步 · 出口消毒：防止模型自己编一条假心声

如果你的项目已经有"剥离模型编造的虚假动作记录"的正则（防止模型说"（我转了账）"却没真调用工具那种），顺手把心声记录也纳入扫描：

FAKE_NARRATION_RE = re.compile(
    r"[（(]\s*(?:心声记录|心声[：:])[^）)]*[）)]"
)
reply = FAKE_NARRATION_RE.sub("", reply)

第六步 · 前端：心声面板

入口放在发送区的附件菜单里（和发图片/表情包并列的那种）：

<div id="thoughtPanel" class="panel-overlay hidden">
  <div class="panel-card">
    <h3>心声</h3>
    <textarea id="thoughtInner" rows="3" placeholder="心里想的…（ta能隐约感知到）"></textarea>
    <textarea id="thoughtSpoken" rows="2" placeholder="说出口的话（可以什么都不说）"></textarea>
    <div class="panel-actions">
      <button id="thoughtCancel">算了</button>
      <button id="thoughtConfirm">发送</button>
    </div>
  </div>
</div>

发送逻辑（心声必填，说出口可留空）：

document.getElementById("thoughtConfirm").addEventListener("click", () => {
  const inner = thoughtInner.value.trim();
  if (!inner) { showToast("心声不能是空的～"); return; }
  const spoken = thoughtSpoken.value.trim();
  closeThoughtPanel();
  send({ thought: inner, spokenOverride: spoken });
});

第七步 · 前端：折叠块渲染

不需要发明新组件——如果你的聊天界面已经有"点开看 AI 这一轮调用了什么工具/想了什么"的折叠条，直接复用同一套视觉语言，只是挂在用户自己这一侧：

function buildThoughtFold(thoughtText) {
  const wrap = document.createElement("div");
  wrap.className = "think-block think-block--user"; // 复用你项目已有的折叠块类名
  const header = document.createElement("div");
  header.className = "think-header";
  header.innerHTML = `<span>💭</span><span>···</span><span class="chevron">›</span>`;
  const body = document.createElement("div");
  body.className = "think-body";
  body.textContent = thoughtText;
  header.addEventListener("click", () => wrap.classList.toggle("open"));
  wrap.append(header, body);
  return wrap;
}

渲染顺序建议：头像 → 折叠块 → 气泡，正好和 AI 那一侧的「头像→折叠块→气泡」左右对称。说出口的话为空时，气泡渲染成一个弱化的「……」而不是空白：

const bubble = document.createElement("div");
if (spoken) {
  bubble.className = "bubble user";
  bubble.textContent = spoken;
} else {
  bubble.className = "bubble user bubble-silent"; // opacity 调低 + 斜体
  bubble.textContent = "……";
}

验收清单

  • 纯心声（不说话）也能发送，不会被"你还没说话呢"拦下
  • 心声 + 说出口同发：模型能感知语气但不会逐字复述心声
  • 刷新页面重载历史，心声折叠块依然正确还原，role 是 user 不是 assistant
  • 引用 + 心声同发：旁白顺序是「引用块 → 心声块 → 说出口的话」
  • 换到对软约束不敏感的模型（如 Gemini 系）连发几条心声，检查回复里不出现"心声记录"字样；出现了就去补强第二条系统提示
  • 消息列表/预览里心声消息显示成 [心声]，不是裸 JSON
03 · 踩过的坑

我们替你们踩过的坑

有的模型会学着历史格式，自己编一条假心声

部分模型（尤其是对软约束遵循度不高、更依赖"历史模式"的模型）看多了"心声记录"这种旁白格式，会在自己的回复里也编一条——比如自己加一句"（心声：其实我也很在意）"。通用感知提示对它没用，必须给这类模型单独追加一条更严格的禁止性约束（见任务书第四步），说清楚这个格式只属于对方，模型自己绝不能用。

忘了把心声纳入"防伪造"正则，会漏网

如果项目里已经有一条"剥离模型编造的虚假动作记录"的正则表达式，很容易漏掉心声这个新格式——上一条坑里模型编出来的假心声记录，最终还是得靠这条正则兜底清理干净。两个机制要配合，缺一个都不够。

引用 + 心声同时存在时，拼接顺序会影响语气

如果聊天还支持"引用某条历史消息回复"，心声和引用同时出现时要注意顺序：引用块在最外层，心声块在其次，说出口的话放在最后——模型是从上到下顺序阅读的，顺序错了会读出很怪的语气。

角色离线/睡着时收到心声，不能把裸 JSON 糊进旁白

如果你的项目有"角色离线时攒消息，回来后一次性看"的机制，攒消息的读取逻辑也要单独处理 __THOUGHT__ 前缀——只透出"说出口的话"，不透出心里想的内容。角色不在场的时候本来就听不到心声，这也顺便避免了把一坨 {"thought": ...} 的裸 JSON 堆进旁白文本里。

心声不要接进和"最近聊了什么"相关的统计/记忆逻辑

如果项目里有额外的"最近互动内容"统计（比如用来驱动角色的某种情绪或偏好引擎），心声不应该被当成"说过的话"计入——它只走这一轮的对话链路，不然会出现"角色因为你心里想的一句话调整了对你的印象"这种説不清楚的副作用。
