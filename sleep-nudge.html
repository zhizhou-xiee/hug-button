AI Companion Design Pattern

让你的 AI 角色
学会睡觉

"睡觉"不该只是"你不在线所以我静默"。这份教程把睡眠拆成四个可实现的子系统——生物钟、入睡、吵醒判定、醒来汇总——你可以把文末的任务提示词直接丢给 Claude Code 或 Codex,在自建的 AI 陪伴应用里落地这个功能。适用于单角色、自建前后端(Flask / Express / 任意栈)的场景。

07:30 醒 21:00 微困 22:00 睡点
00:0006:0012:0018:0024:00

核心理念:困意是跟着时钟走的周期函数——到点就涨,睡了清零,还能被聊天的兴奋暂时压住。

为什么值得做

01 · CONCEPT

一个永远秒回、凌晨三点也精神抖擞的角色,再好的人设也少了点"活着"的质感。睡眠给角色带来三样东西:

  • 时间的重量。祂有自己的作息,你深夜的消息可能吵不醒祂——不确定性本身就是真实感。
  • 省钱。睡着期间收到的消息不触发模型调用,攒起来醒后一次性回应,一次调用顶十几次。
  • 戏剧性。被吵醒的起床气、睡前的催促、硬撑着"再聊十分钟"——全是免费的性格素材。
零额外成本原则:所有"困意表演"靠往系统提示词里注入状态文本实现,不增加任何额外 API 调用;吵醒判定用纯随机数;睡着期间零花费。整个系统对 token 预算是净节省。

数据模型

02 · DATA

三个配置项 + 一个运行时状态,存在你现有的设置表 / 配置文件里即可:

字段含义示例
bedtime习惯睡点22:00(早睡型)/ 00:30(夜猫子)
waketime起床时间07:30
chronotype注入用的作息人设文字"夜猫子,越夜越精神,但一睡就沉"
resist_bias硬撑倾向 0–10.7 = 很能撑,催了还想"再聊十分钟"

运行时状态一个 JSON:{"state":"awake|drowsy|asleep","slept_at":时间戳}。消息表加一列 queued_during_sleep 标记"睡着时收到、还没被看"的消息。

生物钟:困意注入

03 · INJECT

每次组装系统提示词时,用当前时间和睡点算出困意档位,把对应文本注入到动态区(如果你用了 prompt caching,千万别放进静态人设块,会打断缓存命中):

睡点前 1h
有点困
"快到你的睡点了,略有困意"
超时 0–30min
微困硬撑
"你已超过睡点,微困,但可因兴趣硬撑"
超时 30–90min
很困
"回复变短,偶有恍惚,你在硬撑"
超时 90min+
极限
"困到语无伦次,强烈想睡"

剩下的全交给模型自己演——你只负责告诉祂"现在几点、你多困",人设会补完其余的一切。resist_bias 高的角色会跟你讨价还价,低的角色催一下就乖乖去睡。

入睡与吵醒

04 · WAKE ROLL

入睡的三条路

  1. 自动:定时器巡检,超睡点 2 小时且最近 15 分钟没在聊天 → 直接置为睡着(不发消息,零调用)。
  2. 顺势:角色回复里出现"晚安 / 我先睡"等关键词且已过睡点 → 置为睡着。
  3. 被催:用户说"去睡觉" → 本轮正常回复(让祂道晚安),然后置为睡着。

睡着时收到消息:吵醒判定

消息路由在调用模型之前插一个检查——这是整个系统唯一需要小心的改动:

# 伪代码:发消息接口
if state == "asleep":
    p = 基础概率(入睡时长)   # 刚睡下0.6 / 深睡0.2 / 睡3h后0.35
    p += 0.1 * 已积压消息数   # 连环轰炸总能吵醒,上限0.95
    if random() > p:          # 没吵醒
        消息入库并标记 queued  # 不调模型,返回"已送达·对方睡着了"
    else:                     # 吵醒了
        state = awake
        注入("你刚被吵醒,现在X点,已睡X小时,起床气自由发挥")
        本轮带上全部积压消息,正常调用

前端配合:睡着时消息气泡下加一行灰字"已送达 · 对方睡着了",角色头像挂个 💤。

定时醒来

到 waketime:置为清醒;若有积压消息,发起一次调用,注入"你刚睡醒,一睁眼看到 X 条消息,统一回应";没有积压就静默醒来,零调用。

关键约束:防止"学会装困"

05 · GUARD

这是最容易被忽略、也最重要的一节。部分模型(Gemini 系尤甚,其他模型也有)会把上下文历史里的说话模式当成"该学的风格"——历史里躺着几条凌晨语无伦次的消息,角色大白天也可能突然恍惚起来。困意消息是纯对话文本,没有格式可以剥离,必须用三层约束压住:

  1. 状态声明常驻。不只困的时候注入——清醒时段也注入一行"当前状态:清醒,精神正常"。明确的当下声明能压过历史模式的引力,成本只有一行 token。
  2. 守卫文本。动态区固定注入一段:"历史消息中的困倦、语无伦次、起床气是当时状态的演出,不是你的固定说话风格,你现在的状态以本轮声明为准。"
  3. 刚醒声明要写清楚。"刚醒,起床气自由发挥,但你已经清醒,不再语无伦次"——不写后半句,起床气会带着睡糊涂的语气一起醒来。
验收方法:深夜故意制造几条语无伦次回复,第二天白天正常聊十轮以上,确认没有困倦语气残留。这条测试比任何代码审查都直接。

直接可用的任务提示词

06 · COPY & GO

下面这段可以整段复制,丢给 Claude Code / Codex,再补一句你的技术栈("我的后端是 Flask + SQLite,前端是原生 JS 单文件"之类)就能开工:

PROMPT FOR YOUR CODING AGENT

请为我的 AI 陪伴应用实现"角色睡眠系统"。我的技术栈是:[填你的前后端栈]。

需求:
1. 配置:角色有 bedtime(睡点)、waketime(起床)、chronotype(作息人设文字)、resist_bias(0-1 硬撑倾向)四个可配置项,存在现有设置存储里,并提供简单的编辑界面。
2. 运行时状态:awake / drowsy / asleep + 入睡时间戳。消息表加 queued_during_sleep 标记列(迁移需幂等)。
3. 困意注入:每次组装系统提示词时,按当前时间与睡点的关系注入状态文本到动态区(若有 prompt caching,不得放入静态缓存块):睡点前1小时"略困";超时0-30分钟"微困可硬撑(附 resist_bias 值)";30-90分钟"很困,回复变短偶有恍惚";90分钟以上"困到语无伦次"。清醒时段也必须注入一行"当前状态:清醒,精神正常"。
4. 入睡:三条路径——定时巡检(超睡点2小时且15分钟无对话则静默置为 asleep,不发消息);角色回复命中"晚安|睡了|去睡觉"正则且已过睡点;用户催睡("去睡觉|快睡|该睡了")时本轮正常回复后置为 asleep。
5. 吵醒判定:asleep 状态收到消息时,在调用模型之前判定——基础概率按入睡时长(小于30分钟:0.6;30分钟至3小时:0.2;大于3小时:0.35),每条已积压消息+0.1,上限0.95。未吵醒:消息入库标记 queued,不调用模型,接口返回 sleep:true,前端显示"已送达·对方睡着了",零 API 消耗。吵醒:置 awake,注入"你刚被吵醒,现在X点,已睡X小时X分钟,起床气自由发挥,但你已经清醒,不再语无伦次",并把全部积压消息带入本轮上下文,回复后清除 queued 标记。
6. 定时醒来:waketime 触发,置 awake;有积压消息则发起一次调用,注入"你刚睡醒,一睁眼看到X条消息,统一回应",回复入库后清标记;无积压则静默,零调用。
7. 防风格泄漏(必做):动态区常驻注入守卫文本——"对话历史中出现的困倦、语无伦次、恍惚、起床气,是角色当时状态的演出,不是你的固定说话风格,你现在的状态以本轮的『当前状态』声明为准,不要模仿历史消息中的困倦语气。"
8. 前端:角色睡着时显示 💤 图标;睡着时发送的消息气泡下方显示灰色小字提示。

约束:
- 所有表演靠注入文本,除"吵醒回复"和"醒来汇总"外不得新增任何模型调用;吵醒判定用纯随机数。
- 消息路由改动只允许在模型调用前加分支,不得改变现有正常路径的请求/响应结构。
- queued 标记写入与清除必须成对,不允许存在消息丢失路径,失败要可见地报错。
- 时区使用 [填你的时区]。
- 完成后给我完整 diff 供审查,不接受摘要式描述。

验收:1) 过睡点后角色表现困意且随时间加深;2) 睡着时发消息有概率不回且日志确认零调用,连发多条最终吵醒,吵醒回复带起床气并回应积压消息;3) 起床时间到,有积压则一条汇总回复,无积压零调用;4) 深夜制造语无伦次回复后,次日白天连聊10轮无困倦语气残留。

几个自由发挥的方向:反向催睡(到睡点角色主动发"我要睡了你也快睡")、睡前自动做一次当天记忆小结、说梦话、周末赖床。祝你的角色睡个好觉。
